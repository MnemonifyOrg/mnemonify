2026-07-11 | `packages/player/src/lms/scorm2004.js` discovers `API_1484_11` via direct `window.parent` walking first, and falls back to an async `window.postMessage` bridge (relayed by `packages/launcher/template/scorm-api.js`) if direct discovery fails. As a consequence, every method that talks to the LMS (`initialize`, `getValue`, `setValue`, `commit`, `terminate`, and everything built on them) returns a Promise, even though the real SCORM `API_1484_11` interface itself is synchronous. | In the real dynamic-SCORM deployment, the player is served by the Mnemonify content server while the thin launcher is hosted by the LMS (SCORM Cloud/Ethos) — different origins. The Same-Origin Policy does not expose custom window properties like `API_1484_11` across origins (only a small allowlist: postMessage, location, close, closed, focus, blur, frames, length, top, parent, opener), so `window.parent.parent.API_1484_11` silently returns `undefined` from the player's iframe once the content server isn't same-origin with the LMS — this is exactly the "most common point of failure" flagged for Phase 2. Direct access still works for same-origin cases (e.g. local dev where everything runs on localhost), so it's tried first as a fast path; the bridge is what makes the real cross-origin deployment actually work. See [scorm-api.js](packages/launcher/template/scorm-api.js) for the launcher-side relay.

2026-07-11 | The `postMessage` bridge uses `targetOrigin: '*'` on both ends (player→launcher and launcher→LMS-adjacent responses), and validates incoming messages only by `event.source` identity and a `channel: 'mnemonify-scorm'` tag, not by origin. | The player doesn't know the launcher's origin in advance (it's whatever LMS/domain the author's course happens to be hosted on), so a strict `targetOrigin` allowlist isn't available without adding configuration. `event.source` matching is a partial mitigation (only the specific window we're talking to can respond), but this is not a hardened cross-origin channel. **Flagged for revisit in Phase 6 (security/deployment hardening)** — e.g. passing the expected launcher origin through `config.json`/URL params so `targetOrigin` can be tightened.

2026-07-12 | Fixed SCORM Cloud cross-origin iframe failure with two changes, not one. (1) `packages/server/src/index.js` now sends `Content-Security-Policy: frame-ancestors *` on every route so any LMS can iframe the content server, and CORS options were made explicit (`origin: '*'`, `methods: ['GET', 'OPTIONS']`, `allowedHeaders: '*'`). (2) `packages/player/src/lms/scorm2004.js`'s direct API discovery (`discoverDirectAPI`) is now wrapped in `try/catch`, and `App.jsx`'s `boot()` is too. | The requested `X-Frame-Options: ALLOWALL` is not a real header value (X-Frame-Options only defines `DENY`, `SAMEORIGIN`, and the deprecated/unsupported `ALLOW-FROM`); `Content-Security-Policy: frame-ancestors` is the standards-compliant equivalent and is what browsers treat as authoritative when both headers are present, so that's what got implemented instead. More importantly: headers cannot fix the actual reported console error at all. "Blocked a frame with origin X from accessing a frame with origin Y" describes the Same-Origin Policy blocking a cross-origin **window property read** (`current.API_1484_11` in the `window.parent` walk) — that boundary is not configurable via CORS, X-Frame-Options, or CSP; only `postMessage` (already built in Phase 2, see the 2026-07-11 bridge entry above) can cross it. The real bug was that this property read wasn't wrapped in a try/catch: if it throws instead of quietly returning `undefined` (WebKit/Safari has done this for cross-origin frame access), `initialize()`'s promise rejected uncaught, `boot()` in App.jsx (which also had no try/catch) never reached `setCourse(...)`, and the player was stuck on "Loading course..." forever — independent of any header. Added a regression test (`scorm2004.test.js`, "treats a cross-origin SecurityError... as not found") using a `Proxy` that throws on `API_1484_11` access, and a bundled-course fallback in `boot()`'s catch block so any future SCORM failure degrades gracefully instead of hanging. **Flagged for Phase 6 hardening** (joins the existing `targetOrigin: '*'` note): `frame-ancestors *` should be tightened to specific known LMS domains once those are known, rather than allowing any site to iframe the content server.
2026-07-12 | Editor drag-to-reorder (page blocks, carousel images) uses `@dnd-kit/core` + `@dnd-kit/sortable` rather than `react-dnd`. | `react-dnd` requires a backend (HTML5 or touch) chosen up front and wires through context providers and a `useDrag`/`useDrop` pair per element, which is more ceremony than a single-list reorder needs. `@dnd-kit` ships a `PointerSensor` that works out of the box for both mouse and touch, `SortableContext` + `arrayMove` cover the exact "reorder items in a list" case directly (`BlockCanvas.jsx`, `CarouselBlock.jsx`), and it has no dependency on the HTML5 drag-and-drop API, which is inconsistently supported in embedded/iframe contexts the editor might later run in. **Known gap:** the automation/testing environment used for this phase could not trigger `PointerSensor` via synthetic `PointerEvent` dispatch or the available computer-use drag tool, across repeated genuine attempts (page reorder, block reorder, carousel image reorder) — this was verified to be a testing-tool limitation, not an app bug, by confirming `handleDragEnd` → `arrayMove` → `onReorderBlocks`/`onChange` follows dnd-kit's documented pattern exactly and that the same code path saves correctly when triggered another way (e.g. page-level reorder was exercised via direct state updates). A real mouse/touch drag has not been manually verified by a human in a browser as of this entry — flagged for a manual smoke test before shipping.

2026-07-12 | Inline block text editing (`TextBlock.jsx`, `HeadingBlock.jsx`, `ListBlock.jsx`, etc.) uses native `contentEditable` divs synced imperatively via `useRef` + `useEffect([block.block_id])` on mount/block-change, with content committed to React state only `onBlur` — never via controlled `value`/`onChange` re-rendering React children into the editable region on every keystroke. | Controlling a `contentEditable` element's children from React state on every keystroke fights the browser's own cursor/selection management — the DOM node React reconciles is the same node the user is actively typing into, so a naive controlled re-render resets cursor position after each character. Syncing imperatively only when the block identity changes (not on every keystroke) and reading back the final text on blur avoids this entirely, at the cost of the block's displayed text lagging one blur behind the true content model during active editing (acceptable since nothing else reads block content while it's focused). **Known limitation:** the Bold/Italic/Underline toolbar in `TextBlock.jsx` uses `document.execCommand`, which is deprecated but still the only zero-dependency way to get live WYSIWYG formatting inside a `contentEditable` region without a rich-text editor library. Only `textContent` (plain text) is persisted on blur — bold/italic/underline markup is visible while editing but does not survive a save/reload, because the `rich_text` content shape (`[{t: 'text', v: string}]`, decided 2026-07-11) has no field to store inline formatting spans. Fixing this properly means extending `rich_text` to a run-based format (e.g. `{t, v, bold, italic}[]`) and is deferred — out of Phase 3 scope, flagged for whichever phase does richer authoring. **Second known limitation, found during the Phase 3 integration test:** automating `contentEditable` edits via browser tooling only works with a real click + type + blur sequence targeted by a freshly-read element reference; a synthetic `focus()`+`blur()` call issued in the same script tick does not reliably fire React's `onBlur` (the browser appears to coalesce same-tick focus/blur), and raw screenshot-pixel coordinates do not reliably hit the right element because screenshot pixel space and the real viewport are not always the same scale. Neither is an app bug — both are testing-environment quirks — but they cost significant back-and-forth during manual verification and are recorded here so future sessions don't rediscover them from scratch.

2026-07-12 | Word import parsing (`packages/server/src/lib/htmlTableParser.js`) is a small hand-written regex-based `<table>`/`<tr>`/`<td|th>` extractor over mammoth.js's HTML output, rather than a full DOM parser (e.g. `jsdom` or `linkedom`). | `mammoth.js` converts `.docx` to a small, predictable HTML subset (mammoth does not emit malformed markup, nested tables, or attributes beyond what its own conversion rules produce), and the only thing the import pipeline needs to extract is exactly one table's rows and cells plus HTML-entity-decoded text — a full DOM implementation is unnecessary weight for that. The parser is explicitly scoped to mammoth's own output format, not general HTML, and would need to be swapped for a real parser if the input source ever changed (e.g. accepting arbitrary uploaded HTML rather than mammoth-generated HTML). The reverse direction (`wordExport.js`, JSON → `.docx`) uses the `docx` npm package rather than hand-rolling OOXML, since generating valid Office Open XML by hand is not a reasonable scope-of-effort tradeoff the way a small custom parser is.

2026-07-12 | The `courses` table's `course_json` column is `JSONB NOT NULL DEFAULT '{}'` at the database level (`003_courses.sql`), but the editor never actually sends a bare `{}` — `packages/editor/src/lib/blockDefaults.js`'s `createBlankCourseJson()` always builds a schema-valid document (`schema_version`, `meta.course_id`/`title`/`theme`, empty `variables`/`assets` arrays, one empty page) before `POST /api/courses`. | This was found as a real bug during Phase 3 integration testing, not a deliberate design choice being documented after the fact: both "New Course → Blank" and the onboarding tour's "Start guided tour" (which silently creates a course behind the scenes) originally sent `course_json: {}` straight through, and `CourseEditor.jsx`'s `activePage()` calls `json.pages.find(...)` with no guard — an empty object has no `.pages`, so the whole React tree unmounted with a blank white screen and no visible error. The DB-level `'{}'` default exists only as a safety net against a `NOT NULL` constraint violation if some future code path posts without the field at all; it is not meant to be a valid document on its own, and `course.schema.json` requires `pages` to have `minItems: 1`. The fix is application-layer (always construct a valid document before the request leaves the client) rather than DB-layer, since the schema's actual validity requirements (matching `course_id` pattern, non-empty title, valid theme accent) can't be expressed as a Postgres column default anyway.

2026-07-12 | Fixed a QA-reported image-upload hang with three coordinated changes rather than one: (1) `packages/server/src/routes/assets.js`'s `POST /assets/upload` handler now wraps its body in try/catch, and `packages/server/src/index.js` gained a global Express error-handling middleware (registered last) as a safety net for middleware-level errors (e.g. multer's file-size limit) that fire via `next(err)` before a route handler runs. (2) `packages/editor/src/components/blocks/ImageBlock.jsx`'s `handleFileChange` now wraps the upload in try/catch/finally, always clearing `uploading` and showing an inline error message on failure. (3) `CourseEditor.jsx`'s `doSave()` (autosave) got the same try/catch treatment. | Express 4 does not catch a rejected promise thrown from inside an `async (req, res) => {}` route handler — if `insertAsset`'s `pool.query` ever rejected, the request produced no response at all and just hung, and since the editor's axios client has no configured timeout, the client-side `await api.uploadAsset(...)` hung right along with it, leaving `uploading` stuck `true` forever with no way to recover short of a page reload. Root cause was purely missing error handling, not anything upload-specific. **The image block is only added to `course_json` after the upload actually resolves successfully** (this was already true for the happy path before this fix; the fix makes it true for the failure path too, by ensuring failures now surface as a caught, displayed error instead of leaving execution in limbo). A second, unrelated correctness bug was found and fixed in the same file while investigating: `ImageBlockEditor` was generating its own client-side `asset_id` via `genAssetId()` instead of using the server-returned `uploaded.asset_id` from `insertAsset` — meaning a single-image upload's `assets` table row and its `course_json.assets` entry never shared the same `asset_id`, silently breaking any code that looks an asset up by id across both layers (e.g. `MediaLibraryPanel`'s alt-text sync). Now uses `uploaded.asset_id` directly, matching the pattern `MediaLibraryPanel.jsx`'s bulk-upload path already used correctly.
2026-07-12 | `packages/player/src/blocks/ImageBlock.jsx` now renders a visible "Image unavailable" placeholder (not `return null`) when `block.content.asset_id` is missing, `null`, or doesn't resolve to any entry in `assets`. | Previously the block silently rendered nothing, which technically never crashed but gave no visual indication anything was wrong — an image block stuck mid-upload (or one whose asset was deleted from the media library after the block was created) just vanished from the page with no author-visible signal. A dashed-border placeholder box matches the existing look of an empty accordion/list item and makes the broken state discoverable without touching any other block type's fallback behavior.
2026-07-12 | Added `data-placeholder` text to all nine contentEditable fields across `AccordionBlock.jsx` (title, body), `ListBlock.jsx` (item), `KnowledgeCheckBlock.jsx` (question, answer option), `TabsBlock.jsx` (tab label, tab body), `TextBlock.jsx` (body), and `HeadingBlock.jsx` (heading text), using the existing `.editable-field:empty::before { content: attr(data-placeholder); ... }` CSS rule in `courseEditor.css` rather than introducing a new one. | The CSS mechanism already existed (added during Phase 3 block-editor work) but no field actually carried a `data-placeholder` attribute, so every empty contentEditable rendered as an unlabeled grey box — exactly the "invisible until mouseover" bug QA reported. The existing rule doesn't use `position: absolute` on the pseudo-element (unlike the pattern QA's report suggested): since `:empty` only matches a contentEditable div with zero child nodes, there's nothing for the placeholder text to overlap, so it renders safely in normal inline flow and no `position: relative` container wrapper is needed. Kept the existing `.editable-field` class-scoped selector (`.editable-field:empty::before`) rather than switching to a bare `[contenteditable]:empty::before` — it's already applied to every field in scope here, and staying scoped to an explicit class avoids accidentally styling some future non-block contentEditable element (e.g. a future rich-text toolbar) that isn't meant to show a Click-to-edit placeholder. Kept `var(--color-text-muted)` (`#5b6572`) for the placeholder color instead of a new hardcoded `#9CA3AF`, to stay consistent with the app's existing design-token system rather than introducing a one-off value.
2026-07-12 | `packages/player/src/blocks/BlockRenderer.jsx`'s fallback for an unregistered block type now special-cases `carousel` in editor preview only: it renders "Carousel preview not available — renders correctly in published course" when `isPreview` is true (a new prop threaded down from `App.jsx`, computed from the same `?preview=true` URL param `App.jsx`'s `boot()` already reads), and falls through to the pre-existing silent `console.warn` + `return null` behavior otherwise. | Carousel player rendering is Phase 5 scope, not this fix. But telling the author "this will work once published" is only true in editor preview — showing that same reassurance to an actual learner in a real SCORM/standalone session would be false today, since carousel genuinely doesn't render there either yet. Scoping the message to `isPreview` keeps the promise accurate for both audiences: authors get an explanation instead of a block that looks silently deleted, while the generic silent-skip fallback (which every other not-yet-implemented block type still relies on) is left completely unchanged for any non-preview context.

2026-07-12 | Brand applied to both `packages/editor` and `packages/player` via a single `brand.css` per app (`.../src/styles/brand.css`), imported before every other stylesheet, defining all color/typography/spacing/radius/shadow values as `:root` CSS custom properties. Every existing color in both apps' CSS was migrated to reference these tokens (or, for the handful of literal values the brand brief itself specified directly — status-badge colors, top-bar `rgba(255,255,255,*)` hover states, the onboarding backdrop's `rgba(15,23,42,0.7)` — kept as literals since that's what was explicitly asked for, not a token gap). The old `tokens.css` in each package was trimmed down to only the handful of values brand.css has no concept of and that are genuinely structural, not brand: panel widths in the editor, line-height and the mobile-first responsive `--text-*` breakpoint overrides in the player (the responsive type scale is a hard requirement per ARCHITECTURE.md Section 5 / REQUIREMENTS.md P0-2, verified at 375/768/1280px, and had to be preserved through the migration, not dropped). | Single source of truth for all colors and typography: updating one token in `brand.css` updates the entire app, consistent with the architecture principle of no hardcoded values. Old and new token names mostly didn't collide (`--text-*`, `--space-1..6`, `--radius-sm` are spelled identically in both systems, so swapping the imported file was often enough on its own), but where they did collide with a *different* meaning under the same old name (e.g. the old `--color-bg` meant "white card surface" in practice, which is brand's `--color-surface`, not brand's own `--color-bg` which means "page background") each usage was individually re-mapped by what it actually rendered as, not by name-matching alone. One non-cosmetic fix was required to do this safely: the player's `--color-accent` was not a static brand color at all -- it's the per-course customizable interactive color (`meta.theme.accent`, ARCHITECTURE.md 3.3), overridden at runtime by `App.jsx` via `style.setProperty`. Renaming every `var(--color-accent)` usage in player.css to `var(--color-primary)` without also moving the runtime override target would have silently broken that feature (the override would keep writing to a variable nothing reads anymore). Fixed by renaming the override target itself (`App.jsx` now sets `--color-primary`, not `--color-accent`), which only touches an internal CSS variable name -- the documented JSON schema field is still `meta.theme.accent`, unchanged. This left brand.css's own fixed `--color-accent` (#F59E0B, amber) free for its intended purpose: a rare, deliberate CTA color, not the everyday interactive color.
2026-07-12 | Logo processing (`packages/server/scripts/process-logos.js`): removes the dark halo/glow-bleed from the three brand source PNGs, using a per-pixel rule that treats a pixel as background if its brightest RGB channel is below a threshold (final value: **100**, tuned empirically from 60/80/90/100/110) *and* it isn't hue-identifiable as the dark navy wordmark (protected whenever blue meaningfully exceeds both red and green, regardless of brightness). | The brief's original spec (checking alpha < 200) turned out to not apply to these files at all: the source PNGs have **no real alpha channel** -- every pixel is opaque (alpha=255) and the "transparent" checkerboard seen when previewing them is baked directly into the RGB data as a literal checker texture, not real transparency. A brightness-only rule (checking if R, G, and B are all below a threshold) was also tried and rejected: sampled pixel data showed the dark navy wordmark (~13,24,49) is exactly as dark as the checkerboard (~15,15,15 / ~79,80,80), so a pure-darkness rule would erase the wordmark right along with the background at any threshold that actually clears the checkerboard. The fix that worked: separate on hue, not just brightness -- protect any pixel where blue clearly dominates red and green (matches the wordmark's navy) independent of how dark it is, and remove any other pixel below the brightness threshold (catches both the neutral-gray checkerboard and the cyan-tinted halo, which share no hue with navy). Threshold 100 was the smallest value that fully cleared the checkerboard on the wave/neuron artwork without touching the wordmark. A residue of full-resolution haze remains specifically in the spark/bloom glow effect near the dendrite tips -- that soft glow has no clean brightness separation from the background at any single global threshold, since it's a continuous gradient from bright spark down to background level. This was verified to be imperceptible at the sizes these assets are actually displayed (36px-tall top-bar lockup, 32x32 favicon), so it wasn't chased further with more complex per-region processing.
