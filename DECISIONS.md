2026-07-12 | Fixed SCORM Cloud cross-origin iframe failure with two changes, not one. 2026-07-12 | Inline block text editing (`TextBlock.jsx`, `HeadingBlock.jsx`, `ListBlock.jsx`, etc.) uses native `contentEditable` divs synced imperatively via `useRef` + `useEffect([block.block_id])` on mount/block-change, with content committed to React state only `onBlur` â€” never via controlled `value`/`onChange` re-rendering React children into the editable region on every keystroke. | Controlling a `contentEditable` element's children from React state on every keystroke fights the browser's own cursor/selection management â€” the DOM node React reconciles is the same node the user is actively typing into, so a naive controlled re-render resets cursor position after each character. Syncing imperatively only when the block identity changes (not on every keystroke) and reading back the final text on blur avoids this entirely, at the cost of the block's displayed text lagging one blur behind the true content model during active editing (acceptable since nothing else reads block content while it's focused). **Known limitation:** the Bold/Italic/Underline toolbar in `TextBlock.jsx` uses `document.execCommand`, which is deprecated but still the only zero-dependency way to get live WYSIWYG formatting inside a `contentEditable` region without a rich-text editor library. Only `textContent` (plain text) is persisted on blur â€” bold/italic/underline markup is visible while editing but does not survive a save/reload, because the `rich_text` content shape (`[{t: 'text', v: string}]`, decided 2026-07-11) has no field to store inline formatting spans. Fixing this properly means extending `rich_text` to a run-based format (e.g. `{t, v, bold, italic}[]`) and is deferred â€” out of Phase 3 scope, flagged for whichever phase does richer authoring. **Second known limitation, found during the Phase 3 integration test:** automating `contentEditable` edits via browser tooling only works with a real click + type + blur sequence targeted by a freshly-read element reference; a synthetic `focus()`+`blur()` call issued in the same script tick does not reliably fire React's `onBlur` (the browser appears to coalesce same-tick focus/blur), and raw screenshot-pixel coordinates do not reliably hit the right element because screenshot pixel space and the real viewport are not always the same scale. Neither is an app bug â€” both are testing-environment quirks â€” but they cost significant back-and-forth during manual verification and are recorded here so future sessions don't rediscover them from scratch.

2026-07-12 | Logo processing (`packages/server/scripts/process-logos.js`): removes the dark halo/glow-bleed from the three brand source PNGs, using a per-pixel rule that treats a pixel as background if its brightest RGB channel is below a threshold (final value: **100**, tuned empirically from 60/80/90/100/110) *and* it isn't hue-identifiable as the dark navy wordmark (protected whenever blue meaningfully exceeds both red and green, regardless of brightness). | The brief's original spec (checking alpha < 200) turned out to not apply to these files at all: the source PNGs have **no real alpha channel** -- every pixel is opaque (alpha=255) and the "transparent" checkerboard seen when previewing them is baked directly into the RGB data as a literal checker texture, not real transparency. A brightness-only rule (checking if R, G, and B are all below a threshold) was also tried and rejected: sampled pixel data showed the dark navy wordmark (~13,24,49) is exactly as dark as the checkerboard (~15,15,15 / ~79,80,80), so a pure-darkness rule would erase the wordmark right along with the background at any threshold that actually clears the checkerboard. The fix that worked: separate on hue, not just brightness -- protect any pixel where blue clearly dominates red and green (matches the wordmark's navy) independent of how dark it is, and remove any other pixel below the brightness threshold (catches both the neutral-gray checkerboard and the cyan-tinted halo, which share no hue with navy). Threshold 100 was the smallest value that fully cleared the checkerboard on the wave/neuron artwork without touching the wordmark. A residue of full-resolution haze remains specifically in the spark/bloom glow effect near the dendrite tips -- that soft glow has no clean brightness separation from the background at any single global threshold, since it's a continuous gradient from bright spark down to background level.
