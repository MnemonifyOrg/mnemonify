2026-07-11 | Defined concrete `content` JSON shapes for the heading, image, list, tabs, and knowledge-check block types (heading: `{text, level}`; image: `{asset_id}`; list: `{style, items[]}`; tabs: `{items: [{label, body_blocks}]}`; knowledge-check: `{question, options: [{id, text, correct}]}`). ARCHITECTURE.md Section 3.3 only shows worked examples for `text` and `accordion`. | Phase 1 required a hand-written sample course and a schema that validates it, which meant these shapes had to be decided somewhere. Documented here so the schema, player, and future editor/trigger-builder stay consistent with the same content model.

2026-07-11 | Phase 1 trigger engine (`packages/player/src/engine/triggerEngine.js`) fully implements only `SET_VAR` and `ADJUST_VAR`. The other actions in the documented vocabulary (`SHOW_BLOCK`, `HIDE_BLOCK`, `ENABLE_BLOCK`, `DISABLE_BLOCK`, `JUMP_TO_PAGE`, `OPEN_LIGHTBOX`, `SET_STATE`, `SCORM_COMPLETE`, `SCORM_SET_SCORE`) are valid per the schema but log a "not implemented" warning if fired. | Matches REQUIREMENTS.md Section 10 Phase 1 scope: the sample course only exercises an `onOpen` → `SET_VAR` trigger, and the full trigger builder UI/behavior (block visibility, page navigation, lightbox, SCORM reporting) belongs to later phases. Avoids building interaction behavior ahead of the phase that owns it.

2026-07-11 | `packages/player/src/lms/scorm2004.js` discovers `API_1484_11` via direct `window.parent` walking first, and falls back to an async `window.postMessage` bridge (relayed by `packages/launcher/template/scorm-api.js`) if direct discovery fails. As a consequence, every method that talks to the LMS (`initialize`, `getValue`, `setValue`, `commit`, `terminate`, and everything built on them) returns a Promise, even though the real SCORM `API_1484_11` interface itself is synchronous. | In the real dynamic-SCORM deployment, the player is served by the Mnemonify content server while the thin launcher is hosted by the LMS (SCORM Cloud/Ethos) — different origins. The Same-Origin Policy does not expose custom window properties like `API_1484_11` across origins (only a small allowlist: postMessage, location, close, closed, focus, blur, frames, length, top, parent, opener), so `window.parent.parent.API_1484_11` silently returns `undefined` from the player's iframe once the content server isn't same-origin with the LMS — this is exactly the "most common point of failure" flagged for Phase 2. Direct access still works for same-origin cases (e.g. local dev where everything runs on localhost), so it's tried first as a fast path; the bridge is what makes the real cross-origin deployment actually work. See [scorm-api.js](packages/launcher/template/scorm-api.js) for the launcher-side relay.

2026-07-11 | The `postMessage` bridge uses `targetOrigin: '*'` on both ends (player→launcher and launcher→LMS-adjacent responses), and validates incoming messages only by `event.source` identity and a `channel: 'mnemonify-scorm'` tag, not by origin. | The player doesn't know the launcher's origin in advance (it's whatever LMS/domain the author's course happens to be hosted on), so a strict `targetOrigin` allowlist isn't available without adding configuration. `event.source` matching is a partial mitigation (only the specific window we're talking to can respond), but this is not a hardened cross-origin channel. **Flagged for revisit in Phase 6 (security/deployment hardening)** — e.g. passing the expected launcher origin through `config.json`/URL params so `targetOrigin` can be tightened.

2026-07-11 | Phase 2 "page complete" detection is generic and block-type-driven, not course-specific: a page is considered viewed once every `knowledge-check` block on it has been answered (`onCorrect`/`onIncorrect` fired), or immediately if the page has none. | Phase 1/2 has no multi-page navigation, Continue button, or page-visit tracking yet (that's Phase 4's `player chrome` work per REQUIREMENTS.md Section 10). Since the sample course's single page has exactly one knowledge-check block, this makes `completion_rule: "viewed_all_pages"` fire once the learner submits it, and reports a real, non-trivial score/success to SCORM — needed to actually exercise and prove out the score/success/completion reporting path against SCORM Cloud. This heuristic will be replaced by real page-visit tracking (Continue button clicks, linear nav) in Phase 4; nothing here is hardcoded to `sample-course.json`'s specific block IDs or variable names.

2026-07-11 | Added `meta.completion_rule` (`"viewed_all_pages"` \| `"passed_final_quiz"`, optional) to `packages/schema/course.schema.json`, and set it on the sample course. No other ARCHITECTURE.md §3.3 meta fields (`nav_mode`, `page_groups`, `pdf_settings`, `utility_bar`) were added to the schema yet. | Phase 2 explicitly needed this one field ("Add this field to the sample-course.json meta block"); the rest belong to the phases that actually consume them (player chrome in Phase 4, PDF artifact in Phase 5) and adding them now would be schema surface with no corresponding implementation.
